痛點：為什麼 AI 每次開新對話都失憶？

這不是 AI 笨，而是上下文視窗的物理限制

🌊 沙灘上的腳印效應：

AI 助理的「當前對話框」就像沙灘。每當開啟新視窗或海浪一來（對話過長），所有的腳印與記憶都會被無情抹平。

💸 重複溝通的隱性成本：

你每次開新對話都要重複輸入：「我是行銷經理、我們公司的產品是...、請用繁體中文回答」。這不僅消耗耐心，也極大浪費了溝通效率與 API 額度。

底層原理：Token 與 Context Window

關鍵概念 直白解釋 辦公室實體比喻
Token AI 處理文字的「字詞單位」
(1個中文中文字約 1.5~2 Token)		 打字計件的計酬單位
Context Window AI 能一次「同時記住」的字數上限
(例如 8k 至 200k)		 員工辦公桌的桌面大小
(桌上放不下就得丟掉)
上下文塌陷 (Collapse) 當對話超出桌子大小，AI 會自動忘記最開始的設定。 桌面放不下新卷宗，只好把舊資料移到桌底或丟棄
💡 自學應用：每次對話開始時輸入「開工」（Start Work），就是讓 AI 主動把核心背景重新鋪在「辦公桌最顯眼處」，避免失憶！

💡 我們需要一套 「單一真相來源 (SSOT)」 的長效記憶管理機制，將這些固定背景寫下來，讓 AI 自己去讀取，而不是每次都靠你人工輸入。

對話解耦與壓縮寫回心法

別讓 AI 陪你漫無目的地長聊，使用工程思維來管理對話

🎯 心法 1：對話解耦 (一對話一任務)

嚴禁一個對話框做到底。把任務拆開，例如「對話視窗 A 專做規格分析」、「對話視窗 B 專門寫程式碼」。專注能減少背景雜訊，極大提升 AI 輸出的精準度。

📦 心法 2：對話變笨時立即「壓縮寫回」

長對話中 AI 會因為 Context 混亂而開始產生幻覺。此時直接命令 AI：「將目前階段性結論整理成一張 Markdown SSOT 小卡」，接著開啟新對話並丟給它，以乾淨的 Context 繼續高效工作！

省錢大作戰：S / A / B 級模型分工

合理調配模型，能為你節省 80% 的 API 支出：

🏗️ B 級做粗工 (如 DeepSeek, Llama 3)

費用極低或免費。適合進行大量的代碼語法檢索、單純的格式排版美化、撰寫測試檔，或是執行大批量的文本處理命令。

⚡ A 級當主力 (如 GPT-4o-mini, Gemini Flash)

速度極快且性價比極高。適加作為日常主力，負責常規代碼撰寫、Bug 排查、一般文案與文件大綱撰寫。

🧠 S 級做規劃 (如 Claude 3.5 Sonnet, GPT-4o)

邏輯最強但單次費用最昂貴。專供系統架構規劃、複雜業務邏輯重構，以及最終代碼審查與驗收。

為什麼選擇 Obsidian 作為大腦載體？

💾 本地優先

所有筆記均為純文字 Markdown (.md) 檔案，安全存在本地，即使平台倒閉也帶不走。

🔗 網狀雙向連結

使用雙括號 [[Note]] 連結筆記，像大腦神經元一樣建立網絡，拒絕传统死板的分類資料夾。

🤖 AI 讀寫相性極佳

AI 最擅長讀寫結構化的 Markdown。你可以直接授權 Agent 掃描二腦資料夾，實時更新你的知識庫。

🔄 加密自動同步

利用 GitHub 私有倉庫或雲端硬碟，輕易實現多台裝置 (Mac/Win/Mobile) 之間的無縫安全同步。

專案的說明書：CLAUDE.md

如何教導 AI 特助自動遵守專案的工作規矩？

📝 專案裡的「員工守則」

在你的專案根目錄下，建立一個叫 CLAUDE.md 的檔案，詳細記錄該專案的開發指令、程式風格、測試要求與驗收標準。

🧭 開工一秒對齊

當 AI 進入當前目錄，會首先讀取這份說明書。它會立刻知道目前的「角色定位、不准踩的雷區、以及完成定義 (DoD)」，不必你重複說明。

🎯 規則層級優先權 (Context Hierarchy)

由大至小，實現精準的「分權治理」

🌍 全域規則 (Global Context)

定義 AI 特助的通用習慣（如語系偏好、全域指令、安全防火牆限制）。適用於所有專案。

📂 專案規則 (CLAUDE.md)

定義特定專案的專屬風格、指令與驗收標準。當與全域規則衝突時，專案規則擁有優先覆蓋權。

💡 好處：這能讓 AI 在處理不同專案時，自動適配正確的角色設定與商業邏輯，不用人工反覆切換 Prompt。
