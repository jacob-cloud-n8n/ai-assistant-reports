# 🛑 止損三次卡 — 一人版

> 同一個 bug 累計失敗 3 次 → 凍結。不要第 4 次。

---

## 止損規則（兩層架構）

### 第一層：單次派工上限 2 輪

每個 agent 派出去做一件事，最多試 **2 輪**。第 2 輪還失敗就停手，回報卡點。

```
第 1 輪失敗 → 換方法重試
第 2 輪失敗 → 停手，寫凍結報告，交回決策者
```

### 第二層：任務累計上限 3 次

同一個任務不管換了幾個 agent、用了幾種方法，**累計失敗 3 次**就凍結，等 Jacob 拍板。

```
累計第 1 次失敗 → 換方法重試
累計第 2 次失敗 → 升級（換模型/換方法/問人）
累計第 3 次失敗 → 凍結，等 Jacob 決策
```

---

## 為什麼拆兩層？

- **單次 2 輪**：一個 agent 用同一個思路試 2 次還不行，代表這個方向走不通，再試只是燒額度
- **累計 3 次**：允許換人、換策略，但同一個坑踩 3 次代表問題更深，需要人類介入

**2 輪是效率線，3 次是凍結線。**

---

## 升級路徑

| 失敗次數 | 動作 | 範例（以 Claude 系列為例，弱→強） |
|---------|------|------|
| 1 | 換方法重試 | haiku 錯 → 換 sonnet |
| 2 | 升級或問人 | 同模型連錯 2 次 → 換 opus 或問 Jacob |
| 3 | 凍結 | 記錄失敗軌跡 → 等 Jacob 拍板 |

---

## 凍結時要記錄什麼

```
❌ 凍結報告
- 任務：{任務名}
- 嘗試 1：{做了什麼} → {結果}
- 嘗試 2：{做了什麼} → {結果}
- 嘗試 3：{做了什麼} → {結果}
- 卡點：{判斷卡在哪裡}
- 建議：{你認為應該怎麼做}
```

> 📌 兩層口徑對照：**單次派工層**凍結時，報告記錄到「嘗試 2」就停手（同一 agent 同一方向試 2 輪不行就換方向）；**任務累計層**凍結時，報告要記錄到「嘗試 3」（跨 agent／跨方法累計失敗 3 次才真正凍結）。填範本時看你在哪一層，決定要不要填到嘗試 3。

---

## 範例：實際止損流程（任務累計層，完整三次）

```
任務：修復 API 回應格式錯誤

嘗試 1（haiku）：修改 response parser → 仍回 400
→ 換方法重試。

嘗試 2（sonnet）：檢查 request body → 發現缺欄位，加了 → 仍回 400
→ 累計第 2 次失敗，升級：換更強模型並讓 Jacob 一起看卡點。

嘗試 3（opus，Jacob 參與判斷）：比對 API 文件與原始碼，發現欄位已改名 → 改了名稱後仍回 400（另一個未知問題）
→ 累計第 3 次失敗，凍結。記錄：欄位改名不是唯一原因，需要人工重新設計除錯路徑，不再自行嘗試第 4 次。

Jacob 決策：安排時間親自查原始碼與日誌，找到真正的格式錯誤來源後通過。
```

---

## 鐵則

1. **做的人不驗自己**——止損後派 fresh-context 驗（「開新對話、找沒看過過程的 AI 重驗」）
2. **失敗軌跡是升級的門票**——沒有軌跡不准升級
3. **不要「再試一次」**——單次派工第 2 輪失敗就停手，累計第 3 次就凍結

---

> 📌 這份在老師系統裡的真實版本：Jacob 的團隊有「單次派工 2 輪、累計 3 次喊停」的鐵律——同一個 agent 同一個方向最多試 2 輪；同一個任務累計 3 次失敗必須凍結並 Alert Jacob。
