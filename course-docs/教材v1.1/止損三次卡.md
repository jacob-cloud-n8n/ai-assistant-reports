# 🛑 止損三次卡 — 一人版

> 同一個 bug 試了 2 次還修不好 → 停。不要第 3 次。

---

## 止損規則

```
第 1 次失敗 → 換方法重試
第 2 次失敗 → 升級（換模型/換方法/問人）
第 3 次失敗 → 凍結，等 Jacob 決策
```

---

## 為什麼是 2 次不是 3 次？

- 第 1 次：可能是 prompt 沒寫清楚，值得重試
- 第 2 次：如果是同一個方法，代表方法本身有問題
- 第 3 次：如果還在試，只是在燒額度

**2 次是效率線，3 次是浪費線。**

---

## 升級路徑

| 失敗次數 | 動作 | 範例 |
|---------|------|------|
| 1 | 換方法重試 | haiku 錯 → 換 sonnet |
| 2 | 升級或問人 | 同模型連錯 2 次 → 換 opus 或問 Jacob |
| 3 | 凍結 | 記錄失敗軌跡 → 等 Jacob 拍板 |

---

## 凍結時要記錄什麼

```
❌ 凍結報告
- 任務：{任務名}
- 嘗試 1：{做了什麼} → {結果}
- 嘗試 2：{做了什麼} → {結果}
- 卡點：{判斷卡在哪裡}
- 建議：{你認為應該怎麼做}
```

---

## 範例：實際止損流程

```
任務：修復 API 回應格式錯誤

嘗試 1（haiku）：修改 response parser → 仍回 400
嘗試 2（sonnet）：檢查 request body → 發現缺欄位，加了 → 仍回 400
→ 凍結。記錄：API 文件寫的欄位名稱與實際不符，需要查原始碼確認。

Jacob 決策：查原始碼，發現欄位已改名，更新文件後通過。
```

---

## 鐵則

1. **做的人不驗自己**——止損後派 fresh-context 驗
2. **失敗軌跡是升級的門票**——沒有軌跡不准升級
3. **不要「再試一次」**——第 2 次失敗就停

---
