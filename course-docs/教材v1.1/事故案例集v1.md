# 🚨 事故案例集 v1 — 去敏版

> 這些是真實發生過的事故，改寫後去掉敏感資訊。讀完你就知道「哪些事不能做」。

---

## 案例 1：Token 外洩事件

**發生了什麼**：某工程師把 API Key 直接貼進聊天視窗，AI 把它寫進了公開 repo 的 commit 記錄裡。

**後果**：金鑰被掃描工具偵測到，GitHub 發出安全警報。必須立即廢棄舊 Key、產生新 Key，並檢查所有使用該 Key 的服務是否被盜用。

**教訓**：
- API Key 永遠不進 Git（包括 commit message）
- 用 `.env` 存金鑰，`.gitignore` 忽略它
- 交給 AI 前先遮罩：`sk-...abc` → `sk-***abc`

## 案例 2：長對話變笨

**發生了什麼**：一個對話連續聊了 3 天，討論了 5 個不同專案。AI 開始搞混「哪個專案的規則」，給出矛盾的建議。

**後果**：浪費 2 小時debug，最後發現是 AI 把 A 專案的設定套到 B 專案去了。

**教訓**：
- 一個對話只做一件事（對話解耦）
- 變笨時：讓 AI 把精華壓縮寫回本地，開新對話重載
- 重要設定寫進 CLAUDE.md，不只靠對話記憶

## 案例 3：自動化誤觸發

**發生了什麼**：設定好的 n8n 自動化工作流，在測試時觸發了真實的客戶通知。

**後果**：客戶收到奇怪的測試訊息，需要道歉並解釋。

**教訓**：
- 測試用獨立環境，不碰 production
- 自動化 workflow 上線前，你（或負責人）必須拍板
- 「緊急停止」機制：知道怎麼快速關掉自動化

---

## 案例 4：同 Bug 三次喊停

**發生了什麼**：AI 修一個 bug，第一次修錯方向，第二次換方法還是錯，第三次想「再試一次」。

**後果**：燒掉 3 倍額度，問題還在。

**教訓**：
- 同一件事最多試 2 次
- 第 2 次失敗就升級（換模型、換方法、問人）
- 累計 3 次失敗 → 凍結，等 Jacob 決策

---

## 案例 5：部署到錯誤專案

**發生了什麼**：從父目錄跑部署指令，偵測到了錯誤的專案設定，把 A 專案的內容部署到 B 專案去了。

**後果**：B 專案的網站被覆蓋，需要回滾。

**教訓**：
- 部署前確認「我在哪個目錄」
- 用腳本鎖定專案，不跑裸指令
- 部署後立刻驗證 URL

---

## 案例 6：制度自己違反自己

**發生了什麼**：寫了一條規則「路徑先驗證」，結果在同一份文件裡引用了未驗證的路徑。

**後果**：被 fresh-context 的 AI 抓出來，暴露「自律型規則」的脆弱性。

**教訓**：
- 自律型規則 → 寫成腳本自動檢查
- 驗收用機器，不靠「記得要遵守」
- 改完文件跑一次 selfcheck
