# ✅ DoD 檢核卡 — 一人版

> DoD（Definition of Done）=「做完了」的定義。每張任務卡必填，至少一條機器可跑的檢查。

---

## 機器可判定驗收選單

| 交付物類型 | 機器檢查方式 |
|-----------|------------|
| 部署/服務 | `curl` 端點 200 + health payload 比對 |
| 網頁/HTML | 連結存活（`verify-links.sh`）+ html 驗收清單 |
| 程式/腳本 | build/lint/test pass、實跑一次含預期輸出 |
| n8n workflow | fixture 觸發一次 + 預期通知到達 |
| 文件/資料 | 檔案存在 + 關鍵欄位 grep 命中 + 去敏 pattern 零命中 |
| 資料處理 | 筆數/hash 對帳（來源 vs 產出） |

---

## DoD 模板

```
🎯 機器可判定驗收標準：
- [ ] {檔案/服務} 存在且 {具體條件}
- [ ] 跑 `{指令}` 通過
- [ ] grep `{關鍵字}` 命中 ≥ {N} 次
- [ ] 去敏掃描零命中（grep `ghp_|api_key|token=` = 0）
```

---

## 範例：文件任務的 DoD

```
🎯 機器可判定驗收標準：
- [ ] `output/reports/` 目錄存在且 ≥ 3 個 .md 檔
- [ ] 敏感掃描零命中：grep `ghp_|github_pat_|sk-|api_key|token=` = 0
- [ ] 每份報告含「## 驗收人」與「## 日期」欄位
- [ ] 獨立 agent 複驗：內容完整度與格式合規通過
```

**名詞解釋：**
- **獨立 agent 複驗**：開一個新對話、找沒看過過程的 AI 重新驗證，避免「做的人驗自己」自我感覺良好
- **去敏掃描**：用 grep 搜尋敏感字串（API 金鑰、密碼等），確保零泄露

---

## 鐵則

1. **驗收先跑機器檢查**，全過才進語意驗收
2. **做的人不驗自己**——派獨立 agent 驗（開新對話、找沒看過過程的 AI 重驗）
3. **抽象詞禁止**：「完善」「高品質」「適當」一律不行
4. **至少一條機器檢查**——不靠 LLM 語意判斷
