# 💰 省 Token 分工卡 — 一人版

> 核心概念：「機械檢查給腳本，LLM 只做判斷」。開工從 4 分鐘降到 1 分鐘的真實故事。

---

## 什麼是 Token 預算？

- 每次跟 AI 對話都在燒 Token（理解＋生成）
- Token = 錢（按量計費）
- 你的目標：**用最少的 Token 做最多的事**

---

## 分工原則：機器做機器的，AI 做 AI 的

| 工作類型 | 誰做 | 範例 |
|---------|------|------|
| 機械檢查（檔案存在？grep 命中？） | 腳本 | `ls`、`grep`、`md5` |
| 判斷（內容對不對？方向對不對？） | AI | 語意理解、邏輯推理 |
| 搬運（複製貼上、格式轉換） | 腳本 | `sed`、`awk`、`jq` |
| 創造（寫文案、設計方案） | AI | 原創內容 |

---

## 真實故事：開工巡檢從 4 分鐘降到 1 分鐘

### 之前（4 分鐘，15% 額度）

```
AI 讀團隊專區（~200KB）→ 讀規則檔 → 讀任務卡 → 產出摘要
= 75k tokens + 4 分鐘
```

### 之後（1 分鐘，接近零 Token）

```
腳本自動：
1. git fetch + status（機械）
2. 讀板頭今日重點（機械）
3. grep 近 48hr 通知（機械）
4. 找接收卡（機械）
5. 檢查 .tasks/ 逾期（機械）
6. 偵測規則檔變動（機械）

AI 只看摘要，做判斷：
- 今天該做什麼？
- 有沒有需要回應的接收卡？
```

---

## 你怎麼用這個原則？

### Step 1：區分「機械」和「判斷」

問自己：這件事能不能寫成一行指令完成？

- `ls` 能做到 → 機械
- `grep` 能做到 → 機械
- 需要「理解語意」→ AI

### Step 2：把機械的寫成腳本

```bash
# 範例：檢查檔案是否存在
if [ -f "output.md" ]; then echo "✅ 存在"; else echo "❌ 不存在"; fi

# 範例：grep 關鍵字
grep -c "TODO" *.md
```

### Step 3：AI 只看結果做判斷

- 腳本跑完 → 產出摘要
- AI 讀摘要 → 判斷下一步
- 不要讓 AI 做 `ls`/`grep` 能做的事
