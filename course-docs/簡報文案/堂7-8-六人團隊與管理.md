# 簡報文案稿：堂 7-8 六人團隊＋管理者

> M6（實戰案例：六人團隊）＋ M7（從使用者到管理者）

## 第 7 堂：M6 實戰案例 — 六人團隊

### 頁 1：封面
**標題**：實戰案例 — 六人團隊
**要點**：
- 多 Agent 協作的真實運作
- 今天你會畫出你的 Agent 團隊架構圖

### 頁 2：為什麼需要團隊
**標題**：一個人做不久，一個 Agent 做不多
**要點**：
- 單一 Agent 能做的事有限：同時記帳、回客戶、寫文案、管社群——會崩
- 分工的好處：每個 Agent 專注一件事，品質更高、速度更快
- 真實案例：Jacob 從一人單打到六人團隊的過程

### 頁 3：六人團隊架構
**標題**：誰做什麼？
**要點**：
- **Claude**：需求理解與任務拆解——讀懂 Jacob 要什麼，拆成可執行的子任務
- **AG（AntiGravity）**：架構設計與 UI 拋光——只出規格，不碰 code
- **Codex**：技術審查與部署驗收——讀規格、改 code、跑測試、確認上線
- **OpenCode**：執行開發與原型實作——跑指令、做體力活、建立 PR
- **Hermes**：研究、文件與客戶溝通——讀寫文件、發報告、一線陪伴
- **n8n**：自動化中樞——串接服務、觸發工作流、定時推送

### 頁 4：角色分工表
**標題**：六人分工一覽
**要點**：

| Agent | 角色定位 | 允許做 | 禁止做 |
|-------|---------|--------|--------|
| Claude | 需求理解、任務拆解 | 讀代碼、寫任務規格、派工 | 改核心代碼 |
| AG | 架構設計、UI 拋光 | 只讀代碼、寫規格、定義 Sub-agent | 大規模改 code |
| Codex | 技術審查、部署驗收 | 代碼審查、跑測試、合併 PR | 未跑測試就合併 |
| OpenCode | 執行開發、原型實作 | 改 code、跑測試、建 PR | 直接合併 main |
| Hermes | 研究、文件、客戶溝通 | 讀寫 docs、發報告 | 存取核心代碼 |
| n8n | 自動化流程 | 讀 Git、發通知 | 接受 Notion 反向覆寫 |

### 頁 5：今日重點表
**標題**：團隊的 GPS——今日重點表
**要點**：
- 每天開工第一眼看的東西：一張表格，10 秒掌握全局
- 欄位：Agent | 手中任務 | 狀態 | 阻塞於 | 備註
- 狀態標記：✅ 完成 / ] 執行中 / 📋 已指派 / ⏸️ 掛起
- 誰負責更新：Claude 開工和收工時更新
- 範例展示：一張填好的今日重點表

### 頁 6：接收卡
**標題**：派工的正確方式——接收卡
**要點**：
- 派工不能只說「幫我做那個」，要寫接收卡
- 接收卡四要素：收件者（只有一個）+ 一句話描述 + 排優先的行動 + 任務檔路徑
- 一張卡只發一個 agent，不群發
- 範例：一張寫好的接收卡

### 頁 7：狀態機流程
**標題**：任務的生命周期
**要點**：
- 📋 已指派 → ] 執行中 → ✅ 完成
- 中途可能：📤 移交（交給下游）/ ⏸️ 掛起（等外部條件）
- 每個任務只能有一個持有者——看到 ] 或 📤 就停手，不搶
- 禁止：一人多任務並行（同一時間只做一件事）

### 頁 8：DoD 驗收機制
**標題**：做的人不驗自己
**要點**：
- 驗收條件必須是機器可判定的：grep / 跑指令 / curl 200 / 筆數對帳
- 禁止抽象詞：「完善」「高品質」「適當」一律不行
- 驗收流程：機器先跑全過 → 才進語意驗收（開新對話、找沒看過過程的 AI 重驗）
- 範例：一張 DoD 檢核卡

### 頁 9：防呆規則
**標題**：單向指令流
**要點**：
- ✅ Jacob → Agent（指令）
- ✅ Agent → 團隊專區（狀態更新）
- ❌ 直接呼叫其他 Agent（透過團隊專區派發）
- ❌ 未經驗收就宣告完成
- 原則：Git 是唯一真相，Notion 只是鏡子

### 頁 10：Git 協作規則
**標題**：版本控制的紅線
**要點**：
- 每個任務獨立建檔：`.tasks/<task-id>.md`，禁止多人共用
- Commit 訊息遵循 Conventional Commits：`feat:` / `fix:` / `refactor:` / `docs:`
- 絕不 `git add .`——只 stage 相關檔案
- 收工時 stage 好等 AG 統一 commit/push

### 頁 11：安全鐵律
**標題**：不能做的事（紅線）
**要點**：
- 絕不提交：API keys、密碼、Firebase 私鑰、學生真名
- 絕不把 `.env` 放進 repo
- 收工必掃：`git diff` 中搜尋敏感關鍵字
- 筆記：NotebookLM 走瀏覽器 OAuth，不複製 cookie/token

### 頁 12：溝通協議
**標題**：跨團隊通知怎麼寫
**要點**：
- 三級分類：🔴 緊急 / 🟡 需關注 / 🟢 FYI
- 格式：`[日期 發送者 → 收件者] {燈號} {標籤} {摘要}`
- 超過 48hr 沒更新 → Claude 主動追蹤
- 不跨層呼叫：不直接叫其他 agent，透過團隊專區

### 頁 13：一人版 vs 團隊版
**標題**：你現在需要幾個人？
**要點**：
- 一人版：Claude + Hermes，夠用就先這樣
- 三人版：加上 Codex（需要寫 code 時）
- 六人版：完整團隊（有對外產品、有客戶時）
- 漸進式擴展：先求有、再求好

### 頁 14：實做 — 畫架構圖
**標題**：動手做時間
**要點**：
- 繪製你的 Agent 團隊（從一人版開始）
- 定義各 Agent 的角色與權限邊界
- 學員帶走：`團隊專區模板.md`、`狀態機小卡.md`、`接收卡格式.md`

### 頁 15：小結
**標題**：M6 小結——六人團隊的三個關鍵
**要點**：
- **分工**：每個 Agent 專注一件事，品質更高
- **透明**：今日重點表讓所有人 10 秒掌握全局
- **防呆**：狀態機 + 接收卡 + DoD，不靠感覺做事
- 下堂課：從「使用者」升級為「管理者」——你不再只是下指令的人

## 第 8 堂：M7 從使用者到管理者

### 頁 1：封面
**標題**：從使用者到管理者
**要點**：
- 今天你會學派工三件套、Cron 排程、記憶管理、收工報告
- 從「自己做」變成「帶人做」

### 頁 2：管理者的角色轉變
**標題**：你不再是工人，是工頭
**要點**：
- 使用者：自己下指令、自己做、自己驗
- 管理者：拆任務、派工、驗收、記經驗
- 關鍵思維轉變：從「我來做」變成「誰來做」
- 你的價值不再是寫 code，是判斷方向

### 頁 3：派工三件套
**標題**：派工的標準動作
**要點**：
- **接收卡**：一句話說清楚要誰做什麼
- **任務檔**：`.tasks/<task-id>.md`，寫明背景、行動、驗收標準
- **DoD 檢核卡**：至少一條機器可跑的檢查，不靠感覺驗收
- 三件事缺一不可：沒有接收卡 → agent 不知道做什麼；沒有任務檔 → 沒有紀錄；沒有 DoD → 驗收靠吵架

### 頁 4：接收卡實戰
**標題**：怎麼寫一張好的接收卡
**要點**：
- 好的：明確收件者 + 一句話描述 + 排優先行動 + 任務檔路徑
- 壞的：「幫我處理一下」→ agent 會問你要幹嘛，浪費一輪
- 壞的：一張卡發三個 agent → 三人做同一件事
- 練習：把你的下一個任務寫成接收卡

### 頁 5：任務檔實戰
**標題**：怎麼寫任務檔
**要點**：
- 四個必填欄位：任務名 / 背景 / 行動 / 驗收標準
- 行動要可執行：「研究 X」→ 太模糊；「比較 A vs B，列出三差異」→ 可執行
- 驗收標準要機器可判定：`curl 200` / `grep 命中` / `build 通過`
- 範例：一張填好的任務檔

### 頁 6：DoD 實戰
**標題**：怎麼寫 DoD
**要點**：
- 驗收選單：部署→curl 200 / 網頁→連結存活 / 程式→build+test pass / 文件→grep 命中
- 範例：文件任務的 DoD（3 條機器檢查 + 1 條語意驗收）
- 鐵則：做的人不驗自己、抽象詞禁止、至少一條機器檢查
- 練習：為你最近完成的任務補寫 DoD

### 頁 7：Cron 排程
**標題**：讓 AI 自己按時做事
**要點**：
- 定時任務的概念（像鬧鐘：到了時間自動觸發）
- 常見排程：每天 09:00 開工提醒 / 每天 07:00 讀板頭推日報
- 設定方式：n8n Schedule Trigger / 系統 crontab / Agent 內建排程
- 範例：設定一個每日開工提醒

### 頁 8：記憶管理
**標題**：什麼該記、什麼不該記
**要點**：
- 該記：客戶偏好、專案進度、重要決策、踩坑紀錄
- 不該記：臨時草稿、敏感資訊、已過期資料、別人個資
- 記憶清理流程：備份 → 清理 → 驗證（確保重要資料沒被誤刪）
- 安全原則：敏感資料不進記憶體、定期清理過期資訊

### 頁 9：每日開工 SOP
**標題**：開工五步
**要點**：
1. 同步：`git pull` 或讀取團隊專區
2. 讀規則：重讀團隊共識（每次開工必讀，不嫌麻煩）
3. 讀狀態：第一眼看今日重點表（10 秒掌握全局）
4. 看歷史：`git log -n 3` 了解最近變動
5. 回報：向 Jacob 簡短回報今天的打算

### 頁 10：每日收工 SOP
**標題**：收工四步
**要點**：
1. 安全掃描：搜尋敏感關鍵字，確認 .env 不在 status 中
2. 更新筆記：團隊專區狀態 + 學習盤點（今天學到什麼、卡在哪）
3. Git 操作：stage 相關檔案（不 commit，等 AG 統一）
4. 回報：簡短回報今天做了什麼、明天要做什么

### 頁 11：Token 止損線
**標題**：同一個 bug 不要試三次
**要點**：
- 單次派工：一個 agent 同一個方向最多 2 輪，還不行就停
- 累計止損：同一個任務累計 3 次失敗 → 凍結，等 Jacob 拍板
- 凍結時要記錄：做了什麼、結果如何、卡在哪、建議怎麼做
- 範例：修復 API 回應格式錯誤的止損流程

### 頁 12：驗收分級
**標題**：什麼事要問 Jacob、什麼事自己決定
**要點**：
- 🟢 自動道：治理文件、單檔微調、機器驗收通過 → 不需等 Jacob
- 🔴 Jacob 道：對外發布、花錢、簽 token、刪除操作 → 必須 Jacob 拍板
- 分不清時一律當 🔴 處理
- 原則：寧可多問一句，不可擅自做主

### 頁 13：實做 — 設定排程
**標題**：動手做時間
**要點**：
- 為你的 Agent 建立一個 Cron 排程（開工提醒或收工報告）
- 測試自動執行
- 寫一張接收卡 + 一份任務檔 + 一張 DoD
- 學員帶走：`派工三件套模板.md`、`DoD檢核卡.md`

### 頁 14：小結
**標題**：M7 小結——管理者的三個工具
**要點**：
- **派工三件套**：接收卡 + 任務檔 + DoD，確保派得出去、收得回來
- **排程**：讓 AI 自己按時做事，你不用每天記得
- **止損線**：同一個坑不踩三次，保護你的錢包和時間
- 下堂課：規劃你的 AI 團隊藍圖——從痛點出發
