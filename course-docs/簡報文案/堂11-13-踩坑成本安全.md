## 第 11 堂：M10 踩坑紀錄與防呆

### 頁 1：封面
**標題**：踩坑紀錄與防呆
**要點**：
- 這些是真實發生過的事故
- 讀完你就知道「哪些事不能做」

### 頁 2：為什麼要學踩坑
**標題**：別人的学费，你可以免交
**要點**：
- 每一個踩坑案例，都是真金白銀換來的教訓
- 學别人踩过的坑 = 省自己的時間和錢
- 今天的目標：認識六大常見踩坑，學會對應解法

### 頁 3：踩坑 1 — Token 外洩
**標題**：API Key 被偷了
**要點**：
- **發生什麼**：API Key 不小心貼到公開地方（GitHub、論壇、截圖）
- **後果**：別人用你的 Key 燒你的額度（可能一夜花光）
- **解法**：立刻廢棄舊 Key → 產生新 Key → 檢查用量有無異常
- **預防**：`.gitignore` 排除 `.env`、收工必掃 `git diff`

### 頁 4：踩坑 2 — 長對話變笨
**標題**：上下文塌陷
**要點**：
- **發生什麼**：對話太長，AI 開始忘記前面說過的事
- **後果**：回答矛盾、遺漏重要資訊、品質下降
- **解法**：壓縮寫回（把重點寫到檔案）→ 開新對話
- **預防**：重要決策立刻寫入任務檔，不要只存在對話裡

### 頁 5：踩坑 3 — 自動化誤觸發
**標題**：測試搞到 production
**要點**：
- **發生什麼**：在 production 環境測試自動化流程，結果觸發了真動作
- **後果**：自動發送了訊息、修改了資料、搞亂了環境
- **解法**：測試用獨立環境、上線前負責人必須拍板
- **預防**：測試環境和 production 完全隔離

### 頁 6：踩坑 4 — 同 Bug 三次喊停
**標題**：燒額度的無底洞
**要點**：
- **發生什麼**：同一個 bug 換了三種方法還是修不好
- **後果**：燒了大量 API 額度，問題還沒解決
- **解法**：凍結——記錄失敗軌跡，等 Jacob 決策
- **預防**：單次派工 2 輪停手、累計 3 次凍結

### 頁 7：踩坑 5 — 部署到錯誤專案
**標題**：搞混環境
**要點**：
- **發生什麼**：從錯誤的目錄跑部署腳本，deploy 到了不該 deploy 的專案
- **後果**：覆蓋了 production 內容、搞亂了其他專案
- **解法**：確認目錄、用腳本鎖定專案名稱
- **預防**：部署前 double-check 專案 ID、用 `--dry-run` 先跑一次

### 頁 8：踩坑 6 — 制度自己違反自己
**標題**：自律失效
**要點**：
- **發生什麼**：制定了規則但自己沒遵守（例如說好不 `git add .` 但還是加了）
- **後果**：敏感資料被 commit、流程失控
- **解法**：把規則寫成腳本自動檢查（例如 pre-commit hook）
- **預防**：能自動化的規則不要靠人記

### 頁 9：解法速查表
**標題**：六大踩坑 → 六個解法
**要點**：

| 踩坑 | 關鍵解法 | 預防措施 |
|------|---------|---------|
| Token 外洩 | 廢棄舊 Key + 產生新 Key | .gitignore + 收工掃描 |
| 長對話變笨 | 壓縮寫回 + 開新對話 | 重要決策寫入檔案 |
| 自動化誤觸發 | 獨立環境 + 拍板才上線 | 測試/production 隔離 |
| 同 Bug 三次 | 凍結 + 等 Jacob | 2 輪/3 次止損線 |
| 部署錯誤 | 確認目錄 + 鎖定專案 | --dry-run + 雙重確認 |
| 制度違反 | 腳本自動檢查 | pre-commit hook |

### 頁 10：事故案例集
**標題**：真實事故改寫
**要點**：

- 案例 3：同一個 bug 試了 5 次 → 燒了 $30 額度 → 凍結後 10 分鐘找到解法
- 教訓：每一個坑都有跡可循，問題是你要不要學

### 頁 10B：案例深挖 — Token 外洩
**標題**：一分鐘損失 $50 的故事
**要點**：
- **時間線**：14:00 貼 Key 到 GitHub → 14:01 被爬蟲掃到 → 14:30 已被刷 $50
- **怎麼發現的**：收到 API 用量異常警報
- **怎麼處理**：廢棄舊 Key → 產生新 Key → 檢查 Git 歷史 → 確認沒有其他洩漏
- **花了多少時間修**：30 分鐘（但 $50 已經回不來）
- **教訓**：預防 5 分鐘 > 修復 30 分鐘 > 損失 $50

## 第 12 堂：M11 成本優化術

### 頁 1：封面
**標題**：成本優化術
**要點**：
- 今天你會學省錢策略
- 試算你的 AI 月費

### 頁 2：AI 花多少錢
**標題**：成本結構拆解
**要點**：
- **工具費用**：Hermes 免費 / Codex 有限免費 / n8n 自架免費
- **API 費用**：按 Token 計費（最重要的成本來源）
- **部署費用**：Zeabur 免費額度 / 自架伺服器月費
- **隱形成本**：你的時間（這才是最貴的）

### 頁 3：Token 計費原理
**標題**：Token 是什麼？為什麼要省？
**要點**：
- Token ≈ 文字片段（大約 1 個中文字 = 1-2 Token）
- 模型越強（opus > sonnet > haiku），每個 Token 越貴
- 一場對話可能花幾百到幾千 Token
- 一個月下來，可能花幾十到幾百美元

### 頁 4：省錢策略 1 — 機械檢查給腳本
**標題**：零 Token 的做法
**要點**：
- 很多檢查不需要 AI：grep 搜尋、curl 測試、build 通過
- 把這些交給腳本（Shell/Python），不花任何 Token
- AI 只做需要「判斷」的事：內容品質、邏輯正確性
- 原則：**能用腳本做的，不要用 AI**

### 頁 5：省錢策略 2 — 分級使用模型
**標題**：B 級做粗工，S 級做判斷
**要點**：
- **B 級（haiku/便宜模型）**：格式轉換、內容填充、初步整理
- **S 級（opus/聰明模型）**：複雜判斷、架構決策、品質驗收
- 用 haiku 做 10 件事的錢，可能只夠 opus 做 1 件事
- 原則：**先用便宜模型試，不行再升級**

### 頁 6：省錢策略 3 — 開工優化
**標題**：從 4 分鐘到 1 分鐘
**要點**：
- 實測故事：原本開工流程要 4 分鐘（讀一堆檔案、跑同步、等回應）
- 優化後：1 分鐘完成（只讀必要的、跳過耗時同步、先掌握全局再深入）
- 節省的不是時間，是每天累積的 Token
- 原則：**每次對話都要有意義，不要浪費在重複讀取**

### 頁 7：省錢策略 4 — OpenRouter
**標題**：用更便宜的模型
**要點**：
- OpenRouter 是模型聚合平台，提供多種便宜模型
- 適合場景：大量粗加工、格式轉換、初步篩選
- 價格：比直接用 Claude/OpenAI 便宜 30-70%
- 限制：品質可能不穩定，重要任務還是用主流模型

### 頁 8：省錢策略 5 — Ollama 本地模型
**標題**：不花錢的做法
**要點**：
- Ollama 讓你在自己電腦上跑 AI 模型（完全免費）
- 適合場景：隱私敏感、大量測試、學習用途
- 限制：需要好電腦（至少 16GB RAM）、品質不如雲端模型
- 原則：**能本地跑的不要上雲端**

### 頁 9：省錢策略 6 — DeepSeek
**標題**：性價比之王
**要點**：
- DeepSeek 是中國的 AI 模型，價格極低
- 適合場景：大量中文處理、預算極限
- 價格：比 Claude 便宜 10 倍以上
- 限制：中文以外的能力較弱、偶爾不穩定

### 頁 10：省錢策略總覽
**標題**：六大策略一覽
**要點**：

| 策略 | 做法 | 適合場景 | 省多少 |
|------|------|---------|--------|
| 機械檢查給腳本 | grep/curl/build 用腳本 | 驗收檢查 | 100%（零 Token） |
| 分級使用模型 | 粗工用 haiku，判斷用 opus | 所有場景 | 50-70% |
| 開工優化 | 只讀必要的檔案 | 每次開工 | 30-50% |
| OpenRouter | 用便宜模型平台 | 大量粗加工 | 30-70% |
| Ollama 本地 | 自己電腦跑模型 | 隱私/測試 | 100%（免費） |
| DeepSeek | 用中國便宜模型 | 中文處理 | 80-90% |

### 頁 11：試算你的月費
**標題**：你的 AI 要花多少錢？
**要點**：
- 步驟 1：估計你每天跟 AI 對話幾次
- 步驟 2：每次對話平均多少 Token
- 步驟 3：乘以 30 天
- 步驟 4：查模型定價表，算出月費
- 範例：每天 10 次 × 2000 Token × 30 天 × $0.015/1K = $9/月

### 頁 11B：三種使用者的月費
**標題**：你可能花多少？
**要點**：

| 使用者類型 | 每天對話 | 每次 Token | 月費（haiku） | 月費（opus） |
|-----------|---------|-----------|-------------|-------------|
| 輕度（偶爾問） | 3 次 | 500 | $0.7/月 | $7/月 |
| 中度（每天用） | 10 次 | 2000 | $9/月 | $90/月 |
| 重度（整天用） | 30 次 | 5000 | $67.5/月 | $675/月 |

- **建議**：先從輕度開始，確認有用再升級
- **省錢關鍵**：分級使用（粗工用 haiku，判斷用 opus）

### 頁 12：實做 — 試算
**標題**：動手做時間
**要點**：
- 計算你的預期用量
- 找出最省錢的方案
- 制定你的成本優化策略
- 學員帶走：`省token分工卡.md`

### 頁 12B：省錢決策樹
**標題**：怎麼選最省？
**要點**：

```
你需要做什麼？
├─ 格式轉換/整理 → 用腳本（零 Token）
├─ 大量粗加工 → haiku + OpenRouter（$0.25/1M Token）
├─ 需要判斷 → sonnet（$3/1M Token）
└─ 複雜決策 → opus（$15/1M Token）
```

- **原則**：先用便宜的試，不行再升級
- **陷阱**：不要一開始就用 opus 做所有事

### 頁 13：小結
**標題**：M11 小結——省錢的三個原則
**要點**：
- **分級使用**：能用腳本的不 AI，能用便宜模型的不選貴的
- **持續優化**：每次開工都在省 Token，累積效果驚人
- **量力而為**：預算決定你的起步，先解決最痛的點

## 第 13 堂：M12 安全實踐

### 頁 1：封面
**標題**：安全實踐
**要點**：
- 今天你會完成安全檢查清單
- 保護你的 AI 助理和資料

### 頁 2：為什麼安全很重要
**標題**：AI 的便利，也是它的風險
**要點**：
- AI 能做的事越多，洩漏的風險越大
- 一次 API Key 外洩可能損失數百美元
- 一次資料洩漏可能影響你的客戶信任
- 安全不是限制，是保護你自己的投資

### 頁 3：三大安全威脅
**標題**：你可能遇到的風險
**要點**：
- **API Key 外洩**：金鑰被盜用，帳單暴增
- **未授權存取**：別人用你的 AI Bot（燒你的額度）
- **資料洩漏**：敏感資訊被 AI 記住、被別人看到

### 頁 4：威脅 1 — API Key 外洩
**標題**：金鑰被偷了怎麼辦
**要點**：
- **怎麼發生的**：Key 貼到 GitHub/截圖/對話中、.env 沒加到 .gitignore
- **立刻做的事**：廢棄舊 Key → 產生新 Key → 檢查用量異常
- **預防**：.gitignore 排除 .env、收工掃描 git diff、不把 Key 貼到任何公開地方
- **工具**：trufflehog3 自動掃描 Git 歷史中的 Key

### 頁 5：威脅 2 — 未授權存取
**標題**：別人用你的 Bot
**要點**：
- **怎麼發生的**：Bot Token 洩漏、白名單沒設、權限太大
- **立刻做的事**：重置 Token → 更新白名單 → 測試
- **預防**：設定白名單（`ALLOWED_USERS`）、限制 Bot 權限、定期更換 Token
- **原則**：最小權限——只給必要的權限

### 頁 6：威脅 3 — 資料洩漏
**標題**：你的資料被看到了
**要點**：
- **怎麼發生的**：敏感資料存到 AI 記憶、對話被記錄、資料沒加密
- **立刻做的事**：清理 AI 記憶、檢查分享設定
- **預防**：敏感資料不進 AI、.env 不進 Git、定期清理過期資料
- **原則**：資料隔離——敏感的跟不敏感的分開

### 頁 7：三道防線
**標題**：防護措施
**要點**：
- **第一道：白名單**——只有授權的人能用（`ALLOWED_USERS`）
- **第二道：金鑰過濾**——`.gitignore` + 敏感掃描（收工必做）
- **第三道：資料隔離**——`.env` 不進 Git、敏感資料不讓 AI 記

### 頁 8：.gitignore 鐵律
**標題**：什麼不能進 Git
**要點**：
- 必須排除：`.env`、`*.key`、`*.pem`、`credentials.json`、`node_modules/`
- 收工必掃：`git diff` 中搜尋 `api_key`、`token`、`secret`、`password`
- 絕不 `git add .`——只 stage 相關檔案
- 驗證：`git status` 確認 .env 不在清單中

### 頁 9：白名單設定
**標題**：限制誰能用你的 AI
**要點**：
- Hermes：`ALLOWED_USERS=你的TelegramID`
- Discord Bot：`DISCORD_ALLOWED_USERS=你的DiscordID`
- n8n：Webhook 加 API Key 驗證
- 測試：用另一個帳號試 → 被拒絕 = 設定成功

### 頁 10：Token 安全
**標題**：保護你的金鑰
**要點**：
- **產生**：用最小權限（只給必要的 scope）
- **儲存**：存在 .env，不寫在 code 裡
- **使用**：不貼到截圖、不貼到對話、不貼到 GitHub
- **更換**：建議每 3 個月換一次
- **監控**：定期查 API 用量，有異常立刻處理

### 頁 11：資料清理
**標題**：定期清理過期資料
**要點**：
- AI 記憶：太久以前的可能不準確，定期清理
- 對話紀錄：敏感對話及時刪除
- 暫存檔：.tmp、.log 定期清理
- 備份：清理前先備份重要資料

### 頁 12：安全檢查清單
**標題**：你的安全 checklist
**要點**：
- [ ] `.env` 在 `.gitignore` 裡嗎？
- [ ] 收工時掃過 `git diff` 了嗎？
- [ ] 白名單設定了嗎？
- [ ] Token 定期更換了嗎？
- [ ] 敏感資料有跟 AI 說「不要記」嗎？
- [ ] 測試環境跟 production 隔離了嗎？

### 頁 13：常見安全場景
**標題**：遇到這些情況怎麼辦
**要點**：
- **發現 Key 外洩**：立刻廢棄 → 產生新 Key → 檢查用量 → 通知相關人
- **Bot 被別人用了**：重置 Token → 更新白名單 → 檢查對話紀錄
- **資料被 AI 記住了**：要求 AI 清理記憶 → 未來避免敏感資料出現在對話中
- **不確定某件事安不安全**：當作不安全處理，先問再做

### 頁 14：實做 — 安全檢查
**標題**：動手做時間
**要點**：
- 走讀安全清單
- 確認你的設定（.gitignore / 白名單 / Token）
- 建立你的安全 checklist
- 學員帶走：`安全檢查清單.md`

### 頁 15：小結
